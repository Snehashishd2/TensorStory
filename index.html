act 06 — what gets stored

Where activations come from

An activation is an intermediate value saved during the forward pass because the backward pass needs it to compute gradients. The FFN saves 3 tensors, attention saves more — and the attention score matrix is b·n_h·s², which is why memory goes quadratic in sequence length.

act 07 — the budget, by area

Memory as a treemap — area is gigabytes

Toggle precision and watch: fp32 and mixed both cost 16 bytes/param of model state — mixed just rearranges it and adds a master copy. What mixed actually shrinks is the activation block (2-byte, not 4). The dashed line is your GPU; cross it and you OOM.

act 08 — why those sizes

One training step explains the bill

Forward caches activations; backward consumes them into gradients (one per param = model size). The optimizer keeps momentum + variance, so it's biggest.

act 09 — memory over time

Four training steps, profiled

Memory isn’t static. Activations climb through the forward pass, then drain into gradients as backward consumes them. Step 1 is the odd one: activations plateau while the caching allocator preps memory blocks, and optimizer states only appear after the first optimizer call — which is why training can succeed on step 1 and OOM on step 2.

act 10 — trade compute for memory

Activation recomputation

Don’t store the activations from act 06 — keep one checkpoint per layer boundary and replay the forward inside the backward pass. Selective drops just the s×s score matrices (FlashAttention does this natively); full keeps ~2·s·b·h bytes per layer for about ×1.33 compute. Toggle it — the treemap above inherits your choice.

act 11 — split the batch

Gradient accumulation

Slice the global batch into micro-batches: forward/backward each one, sum the gradients, divide by N, then take a single optimizer step. Mathematically identical to the big batch — but activation memory now tracks the micro-batch. The price: N sequential passes per step. No free lunch.

act 12 — see the streams

Profile compute and communication

The PyTorch profiler shows the CPU launching kernels asynchronously while compute and communication run on separate CUDA streams. Run them sequentially and the GPU idles; overlap them and comm hides behind compute. Those micro-batches were independent — they could run in parallel. Time for more GPUs.

the gateway

Llama-3 8B can't even hold its model state on one GPU

At 8.03B parameters, model state alone is ~128 GB (mixed-precision, 16 B/param) — past an 80 GB H100 before a single activation. Inference needs only weights + a KV cache; training needs all four blocks. That gap is why parallelism exists.
